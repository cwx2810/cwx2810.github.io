详解BootLoader加载

1.加电转入BIOS控制

cpu加电后，转入0xFFFFFFF0这一地址，此地址存放的是跳转指令jmpF000:E05B，跳转到BIOS起始点。
输入如下命令启动一个stop的qemu模拟器。

输入x /10i $pc反汇编前十条指令，果然第一条就是ljmp长跳转指令。

2.转入引导扇区

首先，进行硬件自检，而后读取引导扇区到内存的0x7c00地址，将控制权交于这一地址。引导扇区中存储了BootLoader和硬盘分区信息以及两个标志位。

在0x7c00处设置断点，继续运行：

回到qemu模拟器的terminal，输入x /10i $pc反汇编前十条指令，和bootasm.s中的起始代码一致，说明BootLoader确实一开始是加载到0x7c00这一地址的。
那么，BootLoader是如何创建的呢？为什么引导扇区中要有标志位？
通过make -V命令查看BootLoader加载过程不难发现BootLoader产生过程。

1.编译kernel

通过gcc把一系列libs、driver、debug、trap、mm中的.c文件编译成.o目标文件。

2.编译bootblock

通过编译bootasm、bootmain、sign生成目标文件，通过ld将bootasm、bootmain、bootblock的目标文件编译成可执行文件bootblock.out。

3.编译BootLoader

通过dd把kernel和可执行文件.out编译成.img系统镜像文件加载到主引导扇区。

4.关于主引导扇区

主引导扇区为512byte，通过sign文件生成，打开sign.c可以看到，char buf[512];定义了大小，buf[510]=0x55;、buf[511]=0xAA定义了最后两位，有了这两个标志位，才能最终输出build 512 bytes boot sector success!，即成功建立主引导扇区。

3.实模式切换保护模式

实模式：这种模式软件访问物理内存空间不能超过1M，无法发挥80386cpu的4GB以上内存管理能力，而且此时软件和操作系统都直接通过指针访问物理地址，他们没有区别，如果不小心修改了操作系统后果显然是灾难性的。
保护模式：顾名思义就是对操作系统进行保护，通过段机制实现线性地址隔离用户程序对操作系统进行保护。

1.将中断标志位置0，不允许中断，段寄存器也置0。

2.要通过把A20地址从0改为1，使全部32条地址线可用，使得可以访问4G空间。

3.通过建立GDT完成逻辑地址到线性地址的映射。

4.置cr0的PE位为1，长跳转使能保护模式。

5.重新设置段寄存器和堆栈。

这些均在bootasm.s中实现。

4.开启分段机制

进行分段的目的就是保护操作系统，使用户程序不能直接访问物理地址，那么就要新增加一个线性地址，并且将逻辑地址转换为线性地址，线性地址最终转换为物理地址。
要将逻辑地址转换为线性地址，首先要知道他们的构成。
逻辑地址：selector+offset
线性地址：段描述符+offset
转换时，逻辑地址以selector为index在gdt表中找到段描述符，从而完成转换。
段描述符：BaseAddress+Limit+Attributes，其中Attributes中又包含了重要的DPL（段描述符权限）。此外属性中还有G（Granularity粒度），G=0时Limit用字节为单位，G=1时Limit用4k为单位。
Selector：包含RPL请求权限。还包含Table Indicator（表的指示位），TI=0表示用GDT表，TI=1表示用LDT表。
CPL（当前权限）在cs代码段中。

5.读取elf格式的系统镜像（BootLoader）到内存

1.读取扇区

通过bootmain中的readsect函数调用内联汇编读取扇区。

2.加载系统镜像

通过bootmain的readseg函数判断elf的值是不是等于e_magic，就是判断读取的是不是elf文件，而后通过ELF系统镜像文件中的elfhdr和proghdr两个头部描述文件，就可以加载img到内存。
eflhdr描述了总的文件信息，proghdr描述了段的信息。

此时，控制权就交给了内存中的系统镜像。

ssh-connect to host github.com port 22-Connection timed out

在上传博客时报错如下：

ssh: connect to host github.com port 22: Connection timed out

尝试安装最新版hexo、git、nodejs，重建博客，依然失败。
尝试普通的git操作，也无法完成，报错如下：

Unknown SSL protocol error in connection to github.com:443

在stackoverflow上给出的最佳答案是使用代理，可是我本身就在使用香港的代理，更换了美国的服务器后，依然报错，奇怪的是，github网站能够登陆，但是在cmd中github.com却ping不通。
继续google，原来gitbash本身需要代理。
所以更改host文件，首先右键-属性，给user最高权限，而后再文件末尾加上：

192.30.253.113 github.com
192.30.252.131 github.com
185.31.16.185 github.global.ssl.fastly.net
74.125.237.1 dl-ssl.google.com
173.194.127.200 groups.google.com
192.30.252.131 github.com
185.31.16.185 github.global.ssl.fastly.net
74.125.128.95 ajax.googleapis.com

一通代理，再次ping即可成功。
继续上传博客，提示无法用https提交。
尝试生成ssh秘钥：

ssh-keygen -t rsa

在C:/Users/Administrator/.ssh下生成rsa和rsa-pub文件，拷贝pub中的秘钥到github的ssh中。
修改hexo根目录下config文件，修改deploy的repo中的https地址为git形式，可以在github博客repo中直接找到并切换这两个地址，复制过来：

deploy: 
      type: git
      repository: git@github.com:cwx2810/cwx2810.github.io.git
     branch: master

再尝试hexo g、hexo d，成功上传博客。

用xpath获取数据时出现Exception in thread "main" java.lang.NoClassDefFoundError

在使用dom4j解析XML时，可以用xpath查找节点，从而读取数据：

public User load(String username) {
    String path = "/users/user[username='"+username+"']";
    Element e = (Element)ud.selectSingleNode(path);
    if(e==null) return null;
    User u = new User();
    u.setNickname(e.elementText("nickname"));
    u.setPassword(e.elementText("password"));
    u.setUsername(e.elementText("username"));
    return u;
}

然而当我testLoad();后，竟然出现错误：

Exception in thread "main" java.lang.NoClassDefFoundError

原来我只是导入了dom4j-1.6.1,jar，而其实dom4j有个压缩文件dom4j.zip，其中有doc和n多个包。

将这些包都导入后，成功读取！

单例设计模式

单例设计模式就是多个类要同时访问修改一个对象，就是说本来多个类修改同名对象做不到，因为只要一new就是一个新的对象，所以现在想只new一个对象所有类都能修改这一个对象，第一，本类new一个对象并提供能让其他类访问的方式，第二，不让别的类new这个对象时new出新对象，那么new对象必须要初始化，初始化必须要构造函数，那就把构造函数私有化，这样就new不了对象了。

但是在本类中提供的访问函数别的类要调用又不能通过new对象，那就通过类调用，被类调用的必须是静态函数，静态函数只能访问（返回）静态成员，所以本类中new的对象所用的名称也就是地址值也必须是静态的在方法区。其他类调用本类提供的访问方式时把new改成类名.方法。

但是地址值是静态的，调用的方法也是静态的，那为什么不直接调用地址值而要通过方法调用呢？因为调用方法是可控的，可以将地址值私有化，不对外开放，在方法里可以设置if等限制功能，方法public开放。

单例里面还有一种方式，开始在本类里不new对象，即对象=null，而在方法里判断，如果对象为空，则new对象。这种方式叫懒汉式。上面那种常规方式，一上来就new好对象叫饿汉式。虽然懒汉式看起来晚加载，省内存，但是一般还得使用饿汉式，一是调用本类必然会调用对象，因为只有调用对象才能调用其他东西，二是我理解的，饿汉式更在思维上更简单正规。
